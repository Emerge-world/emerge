# EBS Longevity Component Design

**Date:** 2026-03-13
**Status:** Approved

## Goal

Add a 6th component to the Emergent Behaviour Score (EBS) â€” **Longevity** â€” that rewards runs where agents survive longer in absolute terms and where larger populations are sustained. This is compatible with infinite-tick runs where no fixed tick budget exists.

## Problem Statement

The current EBS has five components (Novelty, Utility, Realization, Stability, Autonomy). None of them directly reward survival duration or population size. A run where three agents die at tick 500 and a run where three agents survive to tick 8000 can produce similar EBS scores. Similarly, a cohort with five agents surviving as long as a cohort with three agents scores the same. The new component addresses both gaps.

## Design

### New Component: Longevity

### Accumulator Changes in `_compute`

Two additions to the existing single-pass loop in `EBSBuilder._compute`:

1. **`initial_agents`** â€” read from `run_start` event: `len(p.get("config", {}).get("agent_names", []))`. Default 0.
2. **`alive` field in `state_history`** â€” the `agent_state` payload already includes `alive: bool`; add it to each entry so it can be counted.
3. **`total_ticks` fallback** â€” track `max_tick_seen` throughout the pass; use `run_end.total_ticks` when available, else `max_tick_seen`.

No new event types are required.

All other fields in `ebs.json` are unchanged. The top-level `ebs` value is recomputed using the new weight table.

## Constraints

- No new event types or engine changes required.
