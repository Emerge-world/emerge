# Metrics plan

## 1) Core principle: “event-first” instrumentation

Everything you want to measure (EBS, innovation rates, contradiction rates, exploration, diffusion, etc.) becomes easy if you treat the simulation as an **event stream**.

You already have the right architectural insertion points (per-tick lifecycle: perceive → decide → validate → apply → degrade → record). 
And you already have a precedent for a structured audit system that writes JSONL + meta + summary for A/B comparisons (DEC-008). 

### 1.1 Canonical event schema (JSONL)

Define one canonical JSON object per line with:

**Required fields**

* `run_id` (uuid), `seed`, `tick`, `sim_time` (day/hour), `event_type`
* `agent_id` (nullable for world-level events)
* `payload` (small dict; event-type specific)

**Important design constraint**

* Put *only* stable, metric-relevant fields in `payload`. Everything large/verbose goes elsewhere (see §3).

This schema should also match the future dashboard stream model (tick + events array) so the same data supports replay/visualization later. 

### 1.2 Event types you need (minimum set)

These are sufficient to compute all the metrics we discussed:

**Run**

* `run_start` (config, prompt hashes, model ids)
* `run_end` (final aggregates)

**Agent decision pipeline**

* `agent_perception` (pos, vision_radius, resource_hints summary)
* `agent_decision` (raw_response_ref, parsed_action, parse_ok, action_origin)
* `oracle_resolution` (success, effects, precedent_key, cache_hit, message_code)
* `agent_state` (life/hunger/energy, pos, inventory snapshot or diff)

**Innovation**

* `innovation_attempt` (name, description, requires, produces)
* `innovation_validated` (approved/rejected, category, reason_code)
* `custom_action_executed` (name, result, crafting_event if any)

**Memory**

* `memory_compression_started` (episode_window_ref)
* `memory_compression_result` (learnings[], accepted[], rejected[], contradiction_flags)

**World**

* `resource_consumed` (type, qty, pos)
* `resource_regenerated` (type, qty, pos, reason=dawn) (your world has explicit dawn regen rules) 

**Social / evolution (already in your system)**

* `message_sent`, `item_given`, `skill_taught`, `reproduction` (Phase 3/4 already exists in prompts/logic)  

### 1.3 Action origin tagging is non-negotiable

A bunch of your metrics depend on clean separation:

* base actions vs innovated actions
* custom action executions vs innovation attempts

You already have a known issue where agents may treat `pickup` as an innovation action; that must be eliminated or at least correctly tagged for metrics. 
Also, `pickup` is intended as a base action in the agent system. 

Add `action_origin: "base"|"innovation"|"social"|"evolution"` to decision + oracle events.

## 2) How to organize run outputs so metrics are trivial

You currently have human-readable markdown logs (DEC-006) and a planned move to JSONL/structlog.  
The best structure is:

### 2.1 One run directory = one immutable experiment artifact

Write everything under a single directory:

```
data/runs/<run_id>/
  meta.json
  events.jsonl
  blobs/
    prompts/
    llm_raw/
    snapshots/
  metrics/
    timeseries.parquet (or .jsonl/.csv)
    summary.json
    ebs.json
  llm_digest/
    index.md
    agent_<id>_digest.md
    anomalies.json
  debug_markdown/   (optional; can keep current sim_logger output)
```

#### meta.json (small, critical)

Include:

* git commit hash
* seed, width/height, ticks, agent_count
* model identifiers for agent + oracle (you already log model names in overview) 
* prompt template hashes (DEC-008 already specifies SHA-256 hashes) 
* precedent file used (`data/precedents_{seed}.json`) so runs remain interpretable 

#### events.jsonl (authoritative)

This is the *only* file needed to recompute metrics.

Everything else is either:

* cached convenience outputs (metrics/…)
* LLM-friendly summaries (llm_digest/…)
* debug views (debug_markdown/…)

### 2.2 Keep markdown logs, but make them secondary

Your markdown logs are useful for manual debugging, but they are not stable for analysis (they embed huge prompt blocks and are hard to parse). 
Treat them as “debug renderings” of the event stream.

## 3) Make logs analyzable by LLMs (without token waste)

LLMs do best with:

* **compact, consistent structure**
* **minimal repetition**
* **high signal-to-noise**
* **progressive summarization** (hierarchical compression)

Your current logs often repeat full system prompts and long user prompts per tick. Example: the decision prompt includes time, tile, stats, grid, resources, memory, social panels, family panels. 
That’s great for debugging, terrible for LLM-scale analysis.

### 3.1 Split “events” from “blobs”

In events.jsonl, never inline:

* full system prompt
* full user prompt
* full raw LLM response (if large)
* huge grids

Instead, store them as **content-addressed blobs** and reference them.

Example event:

```json
{
  "event_type": "agent_decision",
  "run_id": "...",
  "tick": 47,
  "agent_id": "Ada",
  "payload": {
    "parsed_action": {"action":"move","direction":"east"},
    "parse_ok": true,
    "action_origin": "base",
    "prompt_ref": "blobs/prompts/prompt_47_Ada.txt",
    "raw_response_ref": "blobs/llm_raw/resp_47_Ada.txt",
    "prompt_sha256": "…"
  }
}
```

This keeps JSONL small and stable, but still allows deep inspection.

### 3.2 Produce LLM “analysis packets” (hierarchical summaries)

Create a post-run process that writes:

**llm_digest/index.md**

* run meta (seed, config, models, tick count)
* high-level outcomes (survivors, deaths, innovations, reproduction, etc.)
* links to per-agent digests

**llm_digest/agent_<id>_digest.md**

* 1–2 pages max
* sections:

  * “Key phases” (exploration phase, exploitation phase, innovation phase)
  * “Innovation timeline”
  * “Critical decisions & failures”
  * “Knowledge learned” + contradictions

**llm_digest/anomalies.json**

* list of “interesting” events for deep review:

  * invalid JSON decisions (LLM integration explicitly anticipates invalid JSON; log them cleanly) 
  * contradiction flags
  * repeated failed actions
  * unusual oracle rulings / new precedents

This is how you make logs **LLM-friendly**: the model reads digests first, then pulls relevant raw blobs only when needed.

### 3.3 Add “message_code” / “reason_code” enums

Instead of storing verbose natural-language oracle messages everywhere, store compact codes plus optional text.

Example:

* `reason_code: "NO_RESOURCE_ADJACENT"`
* `reason_code: "EDGE_BLOCK"`
* `message_code: "INNOVATION_REJECTED_REDUNDANT"`

The verbose text can still exist (for debugging), but metrics and LLM summaries should rely on codes.

## 4) Plan to implement the metrics (mapping metrics → required events)

You asked for a plan to implement “those metrics”. The key is to map each metric to event fields.

### 4.1 EBS components

From the framework we defined, every component becomes a query over events.

**Novelty**

* count `innovation_validated.approved==true`
* innovation attempt count (`innovation_attempt`)
* category diversity from `innovation_validated.category` (Oracle already assigns categories) 
* “structural novelty” is a classification step (see below)

**Utility**

* measure state deltas after innovation:

  * compare `agent_state` windows before/after first use of each innovation
* option value:

  * did innovation enable new event types? (e.g., crafting events, inventory changes)

**Realization**

* approved innovations that later appear in `custom_action_executed`
* success ratio in `oracle_resolution.success`

**Stability**

* invalid decision rate: `agent_decision.parse_ok==false`
* contradiction rate: `memory_compression_result.contradiction_flags`
* false knowledge rate: (requires contradiction detector, below)

**Autonomy**

* proactive actions:

  * “moved toward resource while hunger low” (requires `agent_state` + `agent_perception.resource_hints`)
* environment-contingent innovation:

  * innovation attempts correlated with resource presence / shortages
* subgoals:

  * inferred via “reason_code” patterns or LLM post-hoc classifier (optional)

### 4.2 Structural novelty classifier (lightweight, rule-based)

Add a small post-run classifier that tags innovations into:

* `inventory_enabler`, `recipe_action`, `world_modifying`, `coordination_action`, etc.

This can be:

* heuristic first (based on `requires`/`produces` fields already in your innovation schema) 
* optionally LLM-assisted later (but keep deterministic outputs for research)

### 4.3 Contradiction detector (essential for “Stability”)

Your memory compression prompt explicitly asks for reusable lessons. 
But the system needs a way to flag when a “lesson” contradicts actual events.

Implement this deterministically by:

* tagging semantic facts from events into a “ground truth cache” per run (e.g., `EAT_FRUIT → hunger_delta=-20`, `TREE_HAS_FRUIT observed`, etc.)
* when new learnings are produced, run pattern checks:

  * if learning claims “trees have no food” but `resource_consumed fruit` occurred on tree tiles, flag contradiction

This yields:

* `memory_compression_result.contradiction_flags: [...]`
* `false_knowledge_rate` becomes computable

## 5) Implementation roadmap (PR-sized steps)

This is the cleanest sequence that won’t destabilize the sim:

### PR1 — Canonical events.jsonl emitter

* Add an `EventBus` / observer hook in engine/oracle/agent (your architecture is already modular for this). 
* Emit minimal events: `run_start`, `agent_decision`, `oracle_resolution`, `agent_state`, `run_end`
* Write to `data/runs/<run_id>/events.jsonl`

### PR2 — meta.json + blob references

* Add `meta.json` (seed, config, model ids, prompt hashes as per DEC-008) 
* Store rendered prompts and raw responses as blobs referenced by events

### PR3 — Metrics v0 (timeseries + summary.json)

* Build `metrics/summary.json` and `metrics/timeseries.*` from events
* Implement survival + action mix + innovation counts (many of these are already what W&B logger tracks) 

### PR4 — EBS v1 + innovation classifier

* Implement EBS computation and store `metrics/ebs.json`
* Add structural novelty tags for innovations

### PR5 — Memory contradiction detector + Stability metrics

* Add contradiction flags from learnings vs events
* Compute `false_knowledge_rate`, `contradiction_rate`

### PR6 — LLM digests (llm_digest/)

* Generate run and per-agent digest markdown files + anomalies.json
* Ensure digests reference event ids/ticks so the LLM can “drill down”

### PR7 — Fix action origin edge cases

* Resolve the “pickup treated as innovation” / action-space confusion so metrics remain valid.
