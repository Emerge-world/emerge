How Emerge Measures Population Behavior

This page explains how the simulation turns raw run artifacts into population metrics, time-based trends, and the EBS score. It reads like documentation, but it can also point at real metrics files from a run.

Weights & Biases is an optional observer for live telemetry, not the canonical source of truth for run scoring.

Artifact Source

Start with bundled sample artifacts, or point the page at a real metrics directory from data/runs/<run_id>/metrics.

Data Flow

From Events To Metrics

Every run writes an event stream to data/runs/<run_id>/events.jsonl. The builders then read those persisted artifacts after the run completes and derive summary metrics, per-tick metrics, and EBS component scores.

01 events.jsonl

Canonical record of decisions, outcomes, and agent state.

02 summary.json

Whole-run counts and rates for survival, actions, and innovation.

03 timeseries.jsonl

Tick-by-tick trends for the population over time.

04 ebs.json

Weighted interpretation layer built on top of those artifacts.

Run Summary

Population Metrics

These metrics answer what happened across the whole population by the end of the run: who survived, how cleanly actions resolved, and whether innovation attempts became usable behavior.

Survival rate = final survivors / initial population

Oracle success rate = successful oracle resolutions / total oracle resolutions

Innovation approval rate = approved innovations / innovation attempts

Innovation realization rate = approved innovations later used / approved innovations

Tick Metrics

Whole Run Vs Per-Tick Reading

Whole-run metrics tell you how a run ended. Per-tick metrics tell you how the population changed over time, including alive count, mean hunger, mean energy, actions, deaths, and innovation activity.

Interpretation Layer

EBS Breakdown

EBS is a weighted interpretation layer built on top of the recorded artifacts, not a magic intelligence score. 

EBS =
0.30 * Novelty
+ 0.20 * Utility
+ 0.20 * Realization
+ 0.15 * Stability
+ 0.15 * Autonomy

Novelty calculation

Novelty

Novelty = 100 * (0.40 * approval_rate + 0.30 * category_diversity + 0.30 * structural_originality)

  • approval_rate = approved innovations / innovation attempts
  • category_diversity = distinct categories / 4
  • structural_originality = approved innovations not classified as base_extension / approved innovations

Utility calculation

Utility

Utility = 100 * (0.50 * direct_state_improvement + 0.30 * future_option_value + 0.20 * execution_success_rate)

  • direct_state_improvement = used approved innovations with positive welfare delta / used approved innovations
  • welfare = life + energy - hunger
  • The welfare change is measured over a 5-tick window before and after first use.
  • future_option_value = approved innovations that produce non-stat items / approved innovations
  • execution_success_rate = successful custom actions / total custom_action_executed events

Realization calculation

Realization

Realization = 100 * (0.60 * use_rate + 0.40 * execution_success_rate)

  • use_rate = used approved innovations / approved innovations
  • execution_success_rate = successful custom actions / total custom_action_executed events

Stability calculation

Stability

Stability = clamp(100 - 40 * false_knowledge_rate - 30 * invalid_action_rate, 0, 100)

  • false_knowledge_rate = contradictory learnings / total learnings
  • invalid_action_rate = parse fails / total agent_decision events
  • contradiction_rate is reported separately, but equals false_knowledge_rate in v1

Autonomy calculation

Autonomy

Autonomy = 100 * (0.40 * proactive_resource_acquisition + 0.30 * environment_contingent_innovation)

  • proactive_resource_acquisition = non-hungry moves toward seen resources / total moves
  • environment_contingent_innovation = innovation attempts with hunger > 60 / total innovation attempts
  • self_generated_subgoals is reported as 0.0 and is not included in the current autonomy formula

Trust Notes

Limits And Interpretation

Direct counts are more concrete than heuristic scores. EBS is a structured interpretation layer based on implemented signals, not an absolute truth.

Some sub-signals are intentionally partial. For example, self_generated_subgoals is currently fixed at 0.0 and is not included in the current autonomy formula.

Weights & Biases remains an optional observer rather than the official record for metrics or scoring.
